Features as Statistics in a Toy Model

The previous blog argued that features are better understood as statistics than atoms. A feature is meaningful because of what it does for optimization and behavior, not because it is cleanly named in isolation.

This blog investigates how tokens are represented and distributed in representation space. The model is a basic Transformer trained on a simple function-prediction task over pairs (z, f(x)), where the input token string z takes the form "x=-2.500;y=".

Experiment claim. The model does not merely learn a feature for x. It learns a task-shaped statistic: a representation whose geometry depends on which function it must compute.

The setup

The four tasks are identity, square, sin, and gaussian. For each checkpoint, I sample 1000 evenly spaced values of x, run the prompt, and collect the representation of the last prompt token (the equals sign =) before target generation begins.

\[ h(z) = \text{representation of the final prompt token } (=) \] \[ z\mapsto h(z) \]

I then visualize $h(z)$ with PCA and UMAP, measure distance correlations with both $x$ and $y$, and track how the geometry changes during training. The run uses 3000 steps, with dense early checkpoints because most of the behavioral transition happens quickly.

Why this tests the framework

If features were atoms, one might expect a clean internal variable for x, reused across the tasks. But the statistics view predicts something subtler: the useful internal quantity should depend on the objective. A model trained on square may care about magnitude more than sign. A model trained on sin may care about periodic functional structure. A model trained on identity should preserve the input coordinate more directly.

That is roughly what the results show.

square 85.5% 6.91 $x$: 0.223, $y$: 0.425 A more distributed statistic, closer to output value than signed input.
sin 96.5% 1.40 $x$: 0.618, $y$: 0.694 A compressed curve-like statistic with strong functional ordering.
gaussian 98.1% 3.71 $x$: 0.449, $y$: 0.181 A folded geometry where raw x-bin clustering is not the right summary.

Linear directions are useful but incomplete

PCA is still a good sanity check. Identity becomes readable with a few directions. Sine remains compressed, with the first PCA direction explaining most variance. But square spreads into a higher-dimensional representation, and the UMAP view makes its local neighborhoods easier to inspect.

This matches three cautions from the broader framework:

  1. Residual information. Important signal can live outside the top few linear directions.
  2. Nonlinear structure. Relevant computation can depend on folds, curves, and local neighborhoods.
  3. Relational encoding. Meaning can live in distances among points, not in one named feature alone.

Conclusion

In summary, as training and testing accuracy increase, meaningful clusters emerge in representation space. This supports the claim from the previous blog that we should think in terms of a feature manifold, rather than a single LRH-style feature view.

The key point is that learning does not just create isolated features. It reshapes the geometry of token representations into task-useful structure.
